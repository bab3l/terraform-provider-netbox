# Terraform Provider NetBox 1.0 Readiness Checklist

This document is a concrete working checklist for the remaining work identified during the pre-`1.0` review.

## Scope

- Goal: prepare `terraform-provider-netbox` for a stable `1.0.0` release.
- NetBox support policy: keep support fixed to **NetBox v4.1.11** for `1.0.0`.
- Post-`1.0`: version the provider forward alongside future breaking NetBox API versions.

## Recommended execution order

1. Close release blockers.
2. Freeze provider behavior and document it.
3. Fix CI/release reproducibility.
4. Eliminate known test gaps and skipped coverage.
5. Refresh docs/examples and cut a release candidate.

## Priority model

Use the following priority levels while working the checklist:

- **P0 - Release blocker**: must be complete before any `1.0.0-rc1` cut.
- **P1 - Required for 1.0 confidence**: should be complete before `1.0.0`; may be temporarily deferred only with explicit written waiver.
- **P2 - Strongly recommended**: improves maintainability, contributor confidence, or release safety; can slip only if explicitly moved post-`1.0`.
- **P3 - Nice to have / post-1.0 candidate**: valuable, but not worth delaying `1.0` if the provider contract is otherwise stable.

## Phased roadmap

This is the recommended working plan.

### Phase 0 - Finalize scope and ship criteria

**Priority:** P0

**Goal:** lock the release target so the remaining work is measured against a fixed definition of `1.0`.

**Includes**
- Section 1.2 `Version and support policy alignment`
- Section 4.3 `Create explicit release gates`
- Section 7 `Notes / decisions log`

**Deliverables**
- One agreed version story across repo/docs/examples.
- One agreed Go policy.
- One agreed NetBox support statement.
- Written `1.0` release gates.

**Exit criteria**
- No open questions remain about what `1.0` means operationally.

### Phase 1 - Close hard release blockers

**Priority:** P0

**Goal:** eliminate issues that would make the provider contract misleading or unsafe to freeze.

**Includes**
- Section 1.1 `Provider configuration contract`
- Section 1.3 `Freeze user-visible provider behavior`
- Section 2.1 `CI must run the real unit test suite`
- Section 2.3 `Strengthen provider registration tests`

**Deliverables**
- `insecure` is either correctly implemented or intentionally removed.
- Reference normalization and `custom_fields` semantics are frozen.
- CI runs the same real unit test set used locally.
- Registration tests catch accidental surface regressions.

**Exit criteria**
- No known misleading provider behavior remains.
- CI can be trusted as a baseline gate.

### Phase 2 - Establish 1.0 quality evidence

**Priority:** P1

**Goal:** build confidence that the frozen provider behavior is actually reliable.

**Includes**
- Section 2.2 `Add acceptance testing to automation`
- Section 2.4 `Resolve skipped and incomplete acceptance coverage`
- Section 2.5 `Investigate historical custom-fields instability`
- Section 2.6 `Close remaining unit test gaps`

**Deliverables**
- Automated acceptance smoke coverage.
- A decision and resolution path for every currently skipped acceptance test.
- A stable custom-fields acceptance story.
- Baseline unit coverage for all registered surfaces, or explicit waivers.

**Exit criteria**
- Remaining quality gaps are either fixed or documented as accepted exceptions.

### Phase 3 - Make the release reproducible and supportable

**Priority:** P1

**Goal:** ensure the team can repeatedly build, test, document, and ship the provider without hidden drift.

**Includes**
- Section 3.1 `Fix broken and stale documentation references`
- Section 3.2 `Align docs generation workflow`
- Section 3.3 `Document stable 1.0 behavior explicitly`
- Section 4.1 `Pin toolchain versions`
- Section 4.2 `Harden the release process`

**Deliverables**
- Working documentation links.
- Deterministic docs generation.
- Pinned build/release toolchains.
- A repeatable release runbook.

**Exit criteria**
- A release candidate can be produced from repository automation without hand-wavy steps.

### Phase 4 - Clean up remaining gaps before final cut

**Priority:** P2

**Goal:** reduce avoidable paper cuts and obvious gaps before locking the major version.

**Includes**
- Section 3.4 `Fill example gaps`
- Section 5.1 `Finish helper migrations and remove transitional code`
- Section 5.2 `Reduce drift-prone duplicated metadata`
- Section 5.3 `Review cleanup and test environment rough edges`

**Deliverables**
- Missing examples are filled or intentionally waived.
- Transitional helper migrations are cleaned up.
- Drift-prone support scripts are reduced or validated.
- Cleanup/test noise is low enough that failures are actionable.

**Exit criteria**
- The repo feels stable and supportable for outside users and contributors.

### Phase 5 - Release candidate and final cut

**Priority:** P0

**Goal:** validate that the earlier phases actually produced a shippable `1.0`.

**Includes**
- Section 6 `Suggested 1.0 release candidate checklist`

**Deliverables**
- `1.0.0-rc1`
- final acceptance evidence
- final changelog entry
- final migration/expectations note

**Exit criteria**
- All P0 items are complete.
- All P1 items are complete or explicitly waived.
- The release gates from Phase 0 are satisfied.

| Go support policy | What Go version policy is officially supported? | Align docs and automation to the `go.mod` major/minor line and pin CI/release to the same exact version. |
| CI/release Go versioning | Should CI and release use the same exact Go version? | Yes. |
| NetBox support statement | What exact support statement should ship with `1.0`? | "Supported and tested against NetBox v4.1.11; other versions may work but are not supported for `1.0.0`." |

| Release-candidate policy | Will there be a formal `1.0.0-rc1` tag? | Recommended: yes, if acceptance automation is not yet fully mature; otherwise optional. |
