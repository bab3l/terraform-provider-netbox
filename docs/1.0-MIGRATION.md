# Terraform Provider NetBox 1.0 Migration and Expectations

This note is the short adoption guide for the stable `1.0.x` line.

## Scope

- provider release line: `1.0.x`
- officially supported NetBox version: `v4.1.11`
- audience: users upgrading from recent `0.0.x` releases into the stable line

## What `1.0` means

The `1.0` release is primarily a stability cut.

It freezes the provider's current behavior around:

- reference resolution and normalization
- import expectations
- `custom_fields` ownership semantics
- release and documentation reproducibility

For the full behavioral contract, see [docs/BEHAVIOR-GUARANTEES.md](BEHAVIOR-GUARANTEES.md).

## Main user-visible expectations

### 1. Official support stays fixed to NetBox `v4.1.11`

The `1.0.x` line is supported and tested against NetBox `v4.1.11`.

Other NetBox versions may work, but they are not part of the official `1.0` support promise.

### 2. Reference values may normalize to numeric IDs in state

Many references accept names, slugs, or IDs in configuration.

State remains intentionally stability-oriented, and imported or refreshed state commonly settles on numeric IDs.

If long-term state stability matters more than authoring convenience, prefer IDs in configuration.

### 3. `custom_fields` uses a filter-to-owned model

The `1.0` contract is:

- omitted or `null`: preserve NetBox values but do not manage them
- `[]`: clear all custom fields on the object
- non-empty: manage only the named fields in configuration while preserving unmanaged fields already present in NetBox

Resources and data sources intentionally differ here:

- resources return owned `custom_fields` only
- data sources return all visible custom fields from the API response

### 4. Provider `insecure = true` is real in `1.0`

The provider now applies TLS skip-verification behavior through its configured HTTP client when `insecure = true`.

Use it only for controlled environments such as local testing or trusted internal certificates.

## No large migration is expected from recent `0.0.x`

If you are already on the later `0.0.x` builds used during the readiness effort, `1.0` should feel like a contract freeze rather than a feature reset.

The biggest practical expectation is to treat the already-documented behavior as stable going forward.

## Known `1.0` waivers

These limitations are explicitly accepted for the `1.0` cut:

1. `netbox_cable_termination`
	- acceptance coverage remains waived for `1.0`
	- reason: the `netbox_cable` resource does not currently export created termination IDs, and the data source only supports direct ID lookup

2. `netbox_script`
	- acceptance coverage remains waived for `1.0`
	- reason: NetBox scripts are filesystem-managed server fixtures and cannot be created portably through the API

These are known coverage exceptions, not silent gaps.

## Recommended adoption checklist

- use provider version `~> 1.0.0`
- keep NetBox on `v4.1.11` for supported `1.0.x` use
- prefer numeric IDs when you want the least state churn
- review any `custom_fields` usage against the ownership model above
- use [docs/BEHAVIOR-GUARANTEES.md](BEHAVIOR-GUARANTEES.md) as the canonical contract reference
